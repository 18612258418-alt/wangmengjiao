# 01 · 产品与用户

> 这一篇不讲代码。讲的是"为什么做这个产品、给谁用、面对内测反馈怎么取舍"。
> 没有这层判断力，你会照单全收所有用户反馈，迭代很快失去方向。

---

## 一、产品定位（一句话）

> **AI 记忆**帮大学生把碎片化的"看过 / 划过"，自动转化为可回溯、可复习、可练习的"结构化记忆"。

它不是笔记软件，也不是题库 App。它的独特价值在于这条链：

```
学习时随手批注  →  AI 自动理解你在关注什么  →  生成结构化知识卡  →  顺势衍生出复习/练习/交互
```

竞品（普通笔记、网盘、题库）只做其中一段；我们做的是"从批注那一刻起，帮你把知识沉淀下来"的完整闭环。

---

## 二、目标用户

| 维度 | 画像 |
|------|------|
| 人群 | 在校大学生（本科 / 考研 / 四六级备考） |
| 学科 | 理工科为主（物理 / 数学 / 化学），兼顾外语、人文社科、计算机 |
| 设备 | 重点适配 **iPad**（竖屏 / 横屏都要好用，因为学生常用 iPad 看课件做笔记），其次桌面 |
| 场景 | 看网页资料、翻 PDF 课件、整理课堂笔记、考前突击复习 |
| 痛点 | "学的时候觉得懂了，过几天全忘"——缺少把输入转成长期记忆的低成本工具 |

**判断反馈时反复问自己**：这个反馈是不是来自"备考的大学生在真实学习场景里"？如果是某个用户的小众偏好，可以记下但不必急着做。

---

## 三、核心价值主张（产品的"不可动摇区"）

下面这几条是产品的根基，**改动它们要非常谨慎**，因为它们定义了产品是什么：

1. **批注 / 上传即生成**——用户付出极小（圈一下、传个文件），AI 完成大部分整理工作。任何让用户"多填表单、多做配置"的设计都与此相悖。
2. **结构化而非流水账**——AI 输出的是有标题、有模块、有要点、有关联的结构化卡片，不是一段大白话摘要。
3. **像学长辅导，不像教科书**——所有 AI 文案语气是"亲切的学长学姐用'你'讲解"，这是刻意设计的（见提示词里反复出现的"语气像学长"）。
4. **学完顺势能练**——卡片不是终点，可以一键出题、看思维导图、玩交互演示，形成"学 → 测 → 巩固"的延伸。

## 四、当前功能全景（用户能做什么）

| 功能 | 用户动作 | 产出 |
|------|---------|------|
| 批注演示 | 在网页/课件/笔记三种模拟场景上画批注 | 一张记忆卡 |
| 上传图片 | 侧边栏上传图片 | 一张记忆卡（全图分析） |
| 导入资料 | 上传文件（图片 / PDF / 文本） | 一张记忆卡 |
| 记忆 Feed | 按日期浏览卡片，看每日智能总结 | 浏览 / 复习 |
| 卡片详情 | 点开卡片看深度解析、点蓝色名词下钻 | 深入理解 |
| 全部视图 | 按知识类型 / 关键词标签筛选所有卡片 | 检索 / 回顾 |
| 出习题 | 选中若干卡片 → 生成练习题 | 一套带解析的题 |
| 思维导图 | 选中卡片 → 生成知识结构图 | Mermaid 导图 |
| 难点讲解 | 选中卡片 → 生成难点突破 | 分模块讲解 |
| 交互演示 | 选中卡片 → 生成可调参数的动态演示 | 可运行的小交互 |

> 各功能背后调用的模型和提示词，见 [`03_大模型编排地图`](./03_大模型编排地图.md)。

## 五、内测反馈取舍框架

每周收到大学生反馈后，建议用这个优先级排序，避免被零散意见牵着走：

### 优先做（P0）
- **影响"批注→出卡"主流程能否走通**的问题（卡片生成失败、内容明显错误、加载太慢）。这是产品命脉。
- **AI 输出质量问题**：内容空洞、答非所问、出现明显学科错误、题目有歧义。→ 大概率是**提示词**要调，不是代码。
- **iPad 上的可用性问题**（布局错乱、点不到、面板溢出）。目标设备优先。

### 排期做（P1）
- 已有功能的体验增强（如标签筛选更顺手、详情页排版优化）。
- 高频出现（多个用户都提）的新需求。

### 记录但缓做（P2）
- 单一用户的个性化偏好。
- "锦上添花"的炫酷功能，但偏离核心价值主张的。
- 需要后端数据库才能实现的（如跨设备同步）——当前是纯前端 Demo，没有账号体系，做这类要先评估架构改造成本。

### 判断"是提示词问题还是代码问题"的快捷法
| 现象 | 多半是 |
|------|--------|
| 内容质量、语气、结构、漏字段、答非所问 | **提示词**（去 04 文档） |
| 布局、颜色、点击、动画、加载状态、报错弹窗 | **代码 / UI**（让 Cursor 改） |
| 又慢又超时、出题失败回退占位 | **模型选择 / max_tokens / 超时**（去 05 文档） |

## 六、迭代节奏（沿用现有打法）

```
周一~周四：在 main 分支开发新版本（改提示词 / 加功能 / 修 bug）
          每次改动 push main → 开发版自动部署，自己先验
周五前：  确认稳定后，把 main 合并到 release/demo-v1
          → 内测版地址自动更新 → 发给大学生内测用户
周末/下周一：收集反馈，按上面的框架排优先级，进入下一轮
```

> 具体的 git 操作和发版命令见 [`05_运行部署与运维`](./05_运行部署与运维.md)。建议每轮在 `docs/` 下留一份简短的更新日志（参考已有的 `CHANGELOG.md` / `docs/CHANGELOG_0528.md` 格式），方便下一棒的人和你自己回溯。
