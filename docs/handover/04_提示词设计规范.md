# 04 · 提示词设计规范

> 提示词是这个产品的"设计系统"。它们不是随便写的大白话，而是沉淀了很多次踩坑后的约定。
> 这一篇讲：**这些提示词背后的 6 条设计原则**，以及**你改提示词时必须守的铁律**。
> 配合 [`03_大模型编排地图`](./03_大模型编排地图.md) 一起看（03 告诉你改哪个文件，04 告诉你怎么改不出事）。

---

## 一、6 条提示词设计原则（为什么现有提示词长这样）

### 原则 1：不靠字数，靠"结构 + 质量要素"控制输出

❌ 早期写法："请输出 300 字的概述" —— 模型边写边数字，常在句末硬切，造成截断。

✅ 现在写法：规定**结构**（"输出五个模块，每个模块以【模块名】开头"）+ **每节必须包含的信息要素**（如"易错警告必须含：现象 → 原因 → 避免"）+ "完整成句，不要为长度截断"。

> 看 `src/prompts/index.ts` 顶部的"设计原则（去字数化版本）"注释，这是刻意为之。**你加新提示词时也遵循这条**：与其说"写多少字"，不如说"必须包含哪些要素、分几节"。

### 原则 2：用【模块名】分节，禁用 markdown 符号

提示词普遍要求模型"用【模块名】单独占一行开头，模块之间空一行，不使用 #、*、-、** 等 markdown 符号"。

> 因为前端是自己解析【模块名】来排版的（见 `src/utils/parseSections.ts`、`cardDetailParsing.ts`）。如果模型吐出 markdown，排版会乱。**改提示词时不要放开这个限制。**

### 原则 3：`[[双中括号]]` 是产品的"知识网络"命脉

几乎每个文本提示词都有一段"双中括号铁律"，要求模型把正文里**非当前主题**的学术名词用 `[[名词]]` 包裹。前端会渲染成蓝色可点击链接，点击触发知识点下钻（见 03 文档链路 4）。

规则要点（改提示词时必须保留）：
- 每个模块自然出现 1~2 个 `[[名词]]`；
- 中括号里**只放名词本身**，不带动词/形容词/标点；
- 每个 `[[` 必须有配对的 `]]`；
- **不要包裹当前卡片主题词自身**，要包裹相邻/对照/扩展的概念。

> 这是产品差异化体验的来源。删掉它，下钻功能就没内容可点了。

### 原则 4：结构化数据一律输出严格 JSON + 给出 schema + 自检

凡是要被代码解析的（出卡、出题、交互方案、自测题），提示词都会：
1. 给一份**明确的 JSON schema**（每个字段写清含义和约束）；
2. 强调"**只输出 JSON，不要 markdown 围栏、不要解释文字**"；
3. 末尾加**自检清单**（"questions 数量是否等于 N？JSON 能否被 JSON.parse 解析？"）。

> 即便如此模型仍可能出格，所以有 `src/utils/json.ts` 容错兜底（见 03 文档第五节）。**你改 JSON 结构时，必须同步检查解析代码**（见下面铁律 3）。

### 原则 5：按 `skill`（学习类型）分流命题/讲解策略

内容不是"一套提示词打天下"。系统先判断内容属于 6 种学习类型之一，再用对应策略：

| skill | 含义 | 讲解/命题侧重 |
|-------|------|--------------|
| `theory_concept` | 理论/定义/概念（默认） | 核心原理 + 考点解析 |
| `math_problem` | 数学/物理计算题 | 解题步骤 + 易错陷阱 + 同类题 |
| `language` | 外语/语言学习 | 词汇语法 + 地道用法 + 母语干扰点 |
| `experiment_lab` | 实验/研究方法 | 假设 → 方法 → 结论 + 误差分析 |
| `code_cs` | 代码/算法 | 思路 + 复杂度 + 常见 Bug |
| `literature_essay` | 文献/论述/人文社科 | 论点 + 论证结构 + 批判性问题 |

> 实现见 `src/prompts/index.ts` 的 `getSkillSections()` / `buildSkillDeepPrompt()`、`src/prompts/exercise.ts` 的 `strategyForSkill()`。**改某一类的策略时，注意只改对应分支，别误伤其他学科。**

### 原则 6：语气统一"像学长辅导学弟妹，用'你'"

所有面向用户的文本提示词都强调这个语气。这是产品人格的一部分（见 01 文档价值主张 3）。新增提示词请保持一致。

## 二、改提示词的铁律（违反 = 功能静默崩坏）

> 你大概率改提示词远多于改代码。以下是不踩坑的红线。

### 铁律 1：先想清楚改的是"哪个功能的哪一步"
对照 [`03 文档第三节的总表`](./03_大模型编排地图.md) 定位。多步链路里改错步骤＝白改（例：出题质量问题在第 2 步 `buildQuizPrompt`，不在第 1 步蓝图）。

### 铁律 2：保留输出格式契约
- 是 JSON 的：**字段名、字段数量、嵌套结构不要乱动**。前端是按字段名取值的。
- 是【模块名】分节的：**模块名、模块顺序、个数**别随意改，前端按它解析排版。
- 想加/删字段或模块？→ 触发**铁律 3**。

### 铁律 3：改了输出结构，必须同步改"解析代码"
提示词输出和前端解析是一对"暗合同"。改了一边不改另一边，功能会静默失败（显示空白 / 报错 / 回退占位）。对应关系：

| 提示词输出 | 配套的解析/校验代码 |
|-----------|-------------------|
| 出卡 JSON（doubao/import） | `src/utils/api.ts` `callDoubao()` 末尾的字段校验 + `api/import.ts` `normalizeResult()` |
| 详情页 / 概念页【模块名】文本 | `src/utils/parseSections.ts`、`src/utils/cardDetailParsing.ts` |
| 出题 JSON | `src/features/generate/GenerateDrawer.tsx` 里的 parse 逻辑 |
| 交互方案 JSON | `src/utils/interactiveGeneration.ts` `normalizeInteractionPlan()` |
| 任何含公式的 JSON | `src/utils/json.ts`（容错解析，一般不用动） |

> **不确定怎么同步改？** 这正是让 Cursor 帮忙的好场景——见 [`06 文档`](./06_用AI迭代这个项目.md)，把"我改了提示词的输出结构，请同步更新对应的解析代码"交给它。

### 铁律 4：保留"双中括号"和"语气"约定（原则 3、6）
除非你明确要砍掉知识网络功能，否则别删 `[[名词]]` 那段。

### 铁律 5：改完必须本地实测模型输出
提示词是给模型的，不是给编译器的——**代码不报错 ≠ 输出对**。必须 `vercel dev` 跑真实触发，亲眼看模型这次吐的内容/JSON 对不对。建议多触发几次（模型有随机性）。

## 四、一个完整的"改提示词"示例（演练）

**反馈**：内测用户说"出的题太简单，都是背定义"。

1. **定位**（铁律1 + 03 文档）：出题质量 → 多步链路 → 第 2 步 `buildQuizPrompt`（`src/prompts/exercise.ts`）。
2. **找到要改的地方**：该函数里已有"好题质量标准"7 条，其中第 3 条就是"考理解不考背诵"。说明约束在，但模型没遵守好。
3. **怎么改**（原则1：强化要素而非加字数）：
   - 在【命题要求】里把"禁止纯记忆题"写得更具体（举正反例）；
   - 在难度策略 `strategyForSkill()` 对应学科分支里，强化"必须含推理/计算/辨析"；
   - 在末尾自检清单里加一条"逐题确认：这题不查推理就答不出，对吗？"。
4. **保留契约**（铁律2）：只改文案，不动输出 JSON 的字段结构。
5. **实测**（铁律5）：`vercel dev` 选几张卡出题，看难度是否上来了，多试两次。
6. push → 验稳 → 合并到内测分支发版。

> 这个过程几乎不碰代码，全是"产品 + 提示词设计"判断——正是这个岗位的核心价值。
