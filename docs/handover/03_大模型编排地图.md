# 03 · 大模型编排地图 ⭐（核心文档）

> 这是整套交接里最重要的一篇。读懂它，你就掌握了这个产品的"大脑"。
> 它回答四个问题：**用了哪些模型？哪个功能调哪个提示词？多步骤怎么串？输出格式怎么保证不崩？**

---

## 一、两个模型，分工明确

| 模型 | 厂商 | 干什么 | 比喻 |
|------|------|--------|------|
| **豆包视觉** `doubao-vision-pro-32k` | 火山引擎 Ark | **看图**：把图片/截图里的文字、公式、图表"读"出来并初步结构化 | 眼睛 |
| **DeepSeek** `deepseek-v4-pro` / `-flash` | DeepSeek | **写内容**：把要点精修成讲解、出题、生成代码、写总结 | 大脑+笔 |

> 为什么分两个？因为视觉模型擅长"识别"，但生成长文/结构化内容的质量和速度不如专门的文本模型。所以典型链路是 **豆包先看 → DeepSeek 再写**。
>
> `-pro` 质量高但慢，`-flash` 快但略弱。出题这种要快的用 `-flash`，详情页讲解用默认 `-pro`。具体见下表。

## 二、调用通道（前端怎么连到模型）

前端**不直接**调模型（否则 API Key 会暴露在浏览器里）。所有调用都先经过 `api/` 下的 Vercel 后端代理：

```
前端组件
  │  调用 src/utils/api.ts 里的函数
  ▼
┌─────────────────────────────────────────────┐
│ streamText() / callText() / callTextStreamed() │ ──► POST /api/deepseek ──► DeepSeek
│ callDoubao()                                   │ ──► POST /api/doubao   ──► 豆包视觉
│ (导入弹窗直接 fetch)                            │ ──► POST /api/import   ──► 豆包 或 DeepSeek
└─────────────────────────────────────────────┘
        Key 只存在于服务端，前端拿不到
```

| 前端函数（`src/utils/api.ts`） | 走哪个代理 | 流式? | 典型用途 |
|------|-----------|------|---------|
| `callDoubao(imageDataUrl, hasAnnotations)` | `/api/doubao` | 否 | 批注/上传看图 |
| `streamText(prompt, onChunk, onDone)` | `/api/deepseek` | 是 | 边生成边显示的长文 |
| `callText(prompt, options)` | `/api/deepseek` | 否 | 一次拿完的短结构化输出 |
| `callTextStreamed(prompt, options)` | `/api/deepseek` | 是（内部累积成完整串） | 长文/长代码，避免网关超时 |

## 四、四条"多步编排"链路详解（重点理解）

单步的功能不难，难和精彩的都在多步编排。一共有 4 条，下面逐条拆。

### 链路 1：批注/上传出卡（豆包看图 → DeepSeek 写详情）

```
图片
 │
 ▼ 第1步：callDoubao()  →  /api/doubao  →  豆包视觉
 │   用 api/doubao.ts 的 buildPrompt()
 │   产出 JSON：subjectId(学科)、title、summary、overview、
 │            detailSections(要点)、aiKeyPoints、skill(学习类型)、contentType(笔记/作业)...
 │
 ▼ 第2步：streamText()  →  /api/deepseek  →  DeepSeek pro（流式）
 │   把第1步的 JSON 当"原始素材"喂给 buildDetailPagePrompt()
 │   产出五个模块的详情正文（内容概述/解题思路/易错陷阱/知识关联/学习建议）
 │
 ▼ 拼成一张 CardData，写入本地库，显示在 Feed
```
- **编排代码在**：`src/app/App.tsx` 的 `processImage()`（约 169 行起）。
- **关键细节**：第2步设了 **30 秒超时**，超时就只用第1步的内容降级出卡，不让用户干等；第1步若整个失败，用 `buildFallbackCardPayload()` 出一张"演示内容"占位卡。
- **为什么 skill 字段重要**：第1步判断出的 `skill`（6 选 1：理论概念/数学解题/外语/实验/代码/文献）会决定第2步详情页用哪套"模块结构"（见 `buildDetailPagePrompt` 里的 `getSkillSections`）。改一个、影响一串。

### 链路 2：出习题（先规划蓝图 → 再据蓝图逐题出）

```
选中的卡片
 │
 ▼ 第1步：buildQuizBlueprintPrompt()  →  DeepSeek flash
 │   先不出题，先"规划"：这套题该考哪几个考点、各用什么题型/难度、为什么值得考
 │   产出 JSON：examPoints[]（考点列表）
 │
 ▼ 第2步：buildQuizPrompt(notes, config, blueprint)  →  DeepSeek flash
 │   把蓝图塞进提示词，要求"第 i 题对应第 i 个考点"，逐题命题
 │   产出 JSON：questions[]（题干/选项/答案/解析/知识点）
 │
 ▼ 解析成 QuizQuestion[]，存入"学习记录"
```
- **编排代码在**：`src/features/generate/GenerateDrawer.tsx`（约 300 行起）。
- **为什么要两步**：直接让模型出一套题，往往考点重复、难度失控、有的题没营养。先规划再出题，质量明显更高、更可控——这是产品的一个核心设计点，别随便改成一步。
- **为什么用 flash 不用 pro**：出题对速度敏感（用户在等），flash 12 秒能出完，pro 容易触发网关超时回退占位题。这是踩过坑后的选择（见 git 历史里一连串"出题超时"修复）。

### 链路 3：交互演示（设计交互方案 → 生成可运行代码）

```
选中的卡片
 │
 ▼ 第1步：buildInteractionPlanPrompt()  →  DeepSeek（callText，非流式）
 │   先做"交互教学设计"：这个知识点适不适合做交互？交互什么？
 │   给几个可调参数？画面长什么样（sceneDescription 分镜稿）？
 │   产出 JSON：InteractionPlan（learningGoal/controls/outputs/renderStrategy/sceneDescription...）
 │
 ▼ 第2步：buildInteractiveCodePrompt(notes, plan)  →  DeepSeek pro（流式，12k tokens）
 │   把 plan 当"设计稿"，要求像素级实现成一个可运行的 React 单文件组件
 │   还会附上一段"金标准示例代码"(interactiveFewShots.ts) 让模型照着写
 │   产出：完整的 /App.tsx 源码字符串
 │
 ▼ 在 Sandpack 沙箱里编译运行，学生拖滑块看效果
```
- **编排代码在**：`src/utils/interactiveGeneration.ts`（`generateInteractionPlan` + `generateInteractiveCode`）。
- **为什么要两步**：先定"教学意图和画面"，再写代码，能让生成的交互真正服务于理解，而不是炫酷但没用。`sceneDescription` 那段超详细的"分镜稿要求"就是为了让第2步"一次画对"。
- **代码安全**：生成的代码只允许 `import React`，禁止联网/本地存储；跑在 Sandpack 沙箱里，不会影响主应用。生成后还会 `sanitizeInteractiveCode()` 清洗掉模型可能多嘴的 markdown 围栏。

### 链路 4：知识点下钻（详情页 → 点蓝色名词 → 二级讲解页）

```
详情正文里的 [[麦克斯韦方程组]]
 │（前端渲染成蓝色可点链接）
 ▼ 点击  →  buildConceptQuickPrompt(名词, 来源卡片)  →  DeepSeek（流式）
 │   产出"定义速览 / 核心要点 / 怎么理解"三段
 ▼ 二级页里又可以继续点新的 [[名词]]，无限下钻
```
- **编排代码在**：`src/components/ConceptPage.tsx`。
- **关键机制**：所有提示词都被要求在正文里用 `[[名词]]` 包裹关联概念，前端把它渲染成可点击蓝链。这是产品"知识网络"体验的来源，几乎每个提示词里都有"双中括号铁律"，**改提示词时千万别破坏这个约定**（详见 04 文档）。

## 五、输出格式怎么保证不崩（JSON 容错）

模型返回的"JSON"经常不干净：包了 markdown 围栏、前后有废话、或在字符串里写 LaTeX（`$\theta$`、`\frac`）导致非法转义。项目用三层防御保证不崩：

| 防御层 | 在哪 | 做什么 |
|--------|------|--------|
| 抽取 | `src/utils/json.ts` → `extractJsonObject()` | 从一堆文本里抠出最外层 `{...}` |
| 修复 | `src/utils/json.ts` → `repairJsonEscapes()` | 把 LaTeX 反斜杠等非法转义补成合法的，避免 `JSON.parse` 报错 |
| 兜底解析 | `src/utils/json.ts` → `parseJsonLoose()` | 先正常解析，失败就修复后再解析 |

> **给你的启示**：正因为有这层容错，提示词里才敢让模型在 JSON 里写公式。但如果你大改输出结构、或让模型输出非 JSON 的奇怪格式，可能绕过这层防御导致功能"静默失败"（回退到占位内容、报错）。改输出格式前，先看 04 文档的"改提示词铁律"。

## 六、改某个功能的 AI 行为时，按这个流程走

1. 在**第三节的总对照表**里找到功能，确认是**单步还是多步**、调的是**哪个提示词函数**。
2. 多步的话，想清楚要改的是**哪一步**（比如"出题质量差"是改第2步 `buildQuizPrompt` 的质量标准，不是改第1步蓝图）。
3. 打开对应文件的对应函数，按 [`04_提示词设计规范`](./04_提示词设计规范.md) 的铁律修改文案。
4. 本地 `vercel dev` 跑一遍真实触发，看输出对不对（光看代码不算数，模型行为要实测）。
5. 没问题再 push。发版流程见 [`05_运行部署与运维`](./05_运行部署与运维.md)。
