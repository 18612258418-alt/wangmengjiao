# 06 · 用 AI（Cursor）迭代这个项目

> 这一篇是给"懂设计和大模型、但代码没那么深"的你写的工作手册。
> 核心思路：**你做产品判断和提示词设计，让 Cursor 当你的工程师去落地代码。**
> 关键是学会"怎么把需求讲清楚、怎么验证、怎么不把项目改坏"。

---

## 一、心态：你是"设计总监 + 产品经理"，Cursor 是"工程师"

你不需要会写 React。你需要会：
1. **描述清楚要什么**（产品/交互/提示词层面的判断）；
2. **审阅 Cursor 的改动**（看它改了哪、跑起来对不对）；
3. **守住红线**（别让它动不该动的东西）。

> 对照分工：提示词和编排的**设计**是你的活（03、04 文档）；把设计**翻译成代码**是 Cursor 的活。

---

## 二、给 Cursor 提需求的正确姿势

### 模板：背景 + 现象 + 期望 + 约束 + 定位

❌ 差的提问："出题功能不好，优化一下。"（太空泛，它会乱猜）

✅ 好的提问：
> "**背景**：这是一个面向大学生的学习 App，出习题功能走的是'先组卷蓝图再出题'两步链路。
> **现象**：内测用户反馈出的题太简单，大多是背定义。
> **期望**：题目要考理解、推理、辨析，不要纯记忆。
> **约束**：只改提示词文案，不要改输出的 JSON 结构；保留现有的两步编排。
> **定位**：我觉得要改的是 `src/prompts/exercise.ts` 里的 `buildQuizPrompt`，你确认下并修改。"

> 你能写出"定位"这一行，靠的是读过 [`03 文档`](./03_大模型编排地图.md) 的总对照表。这就是那张表的价值。

### 善用 `@` 引用文件
在 Cursor 里输入 `@` 可以直接引用文件（如 `@src/prompts/exercise.ts`），让它聚焦在对的地方，别全仓库乱翻。

### 让它先解释、再动手
不确定时先问："这个功能现在是怎么实现的？涉及哪些文件？" 理解了再让它改。本项目可以让它**先读 `docs/handover/` 这套文档**再干活。

---

## 三、四类高频任务怎么交给 Cursor

### A. 改提示词（你最常做的）
- 你定文案方向，让 Cursor 落到具体函数；
- **务必带上 04 文档的铁律**，例如："如果你改了输出的 JSON 字段，请同步更新对应的解析代码（`callDoubao` 校验 / `normalizeResult` / `parseSections` 等）。"
- 改完让它**总结改了哪些文件、哪些行**，你核对。

### B. 改 UI / 交互 / 修小 bug
- 用 02 文档的"界面↔文件对照表"告诉它去哪个文件；
- 描述要"视觉化"："这个面板在 iPad 竖屏会溢出，请让它在小屏自适应不溢出"。
- 改完一定本地 `vercel dev` 在**真实尺寸**下看（尤其 iPad）。

### C. 加新功能
- 先让 Cursor 出一个**简短方案**（涉及哪些文件、要不要新提示词、走哪个模型几步），你审过再让它写。
- 新功能若要调模型，提醒它复用 `src/utils/api.ts` 的现有通道（`streamText`/`callText`），别另起炉灶把 Key 写到前端。

### D. 排查"AI 生成失败/质量差"
- 先判断是**提示词问题**还是**超时问题**（01 文档的快捷判断表 + 05 文档超时章节）；
- 让 Cursor 一起看：把现象描述清楚，让它检查对应链路的提示词和解析、以及 `max_tokens`/模型/是否流式。

---

## 四、安全红线（务必让 Cursor 也遵守）

把这几条当成"交给 Cursor 时的固定叮嘱"：

1. **不要把 API Key 写进前端代码**（任何 `src/` 里）。Key 只能在服务端 `api/*` 通过 `process.env` 读。带 `VITE_` 前缀的 Key 会被打包暴露。
2. **不要改 `api/` 代理的安全结构**（除非你清楚在做什么）——它的存在就是为了不让 Key 泄露。
3. **改提示词输出结构 → 必须同步改解析代码**（04 文档铁律 3）。
4. **不要提交 `.env.local`**，不要把密钥贴进对话。
5. **大改动先在 `main` 验，别直接动 `release/demo-v1`**（内测用户在用）。

---

## 五、每次改动的验证清单（养成习惯）

改完别急着 push，过一遍：

- [ ] `vercel dev` 本地跑起来了，没报错
- [ ] **亲手触发**了改动相关的功能（不是只看代码）
- [ ] 涉及 AI 的，**多触发两三次**确认稳定（模型有随机性）
- [ ] iPad / 小屏相关的，在窄窗口看过没溢出
- [ ] 没有把 Key 或 `.env.local` 带进改动
- [ ] 让 Cursor 总结了"改了哪些文件"，我大致核对过

通过后再：`git add` → `commit`（写清这次改了啥）→ `push origin main`。发版给内测见 [`05 文档`](./05_运行部署与运维.md) 第四节。

## 六、推荐的每周工作流（把所有文档串起来）

```
1. 收反馈  →  用 01 文档的优先级框架排序
2. 选定本周要做的  →  用 01 的"提示词 vs 代码 vs 超时"快捷判断分类
3. 定位   →  用 02（界面↔文件）/ 03（功能↔提示词）找到要动的地方
4. 设计   →  提示词类按 04 文档自己设计文案；功能/UI 类想清交互
5. 落地   →  本篇(06)的提问模板把活交给 Cursor
6. 验证   →  本篇第五节清单 + vercel dev 实测
7. 发版   →  05 文档：push main 验稳 → 合并 release/demo-v1 → 发内测
8. 记录   →  在 docs/ 留一份简短更新日志（参考 CHANGELOG 格式）
9. 进入下一轮
```

---

## 七、给你的一句话

这个项目把"难复制的判断（产品 + 提示词）"和"易生成的实现（代码）"分开了。**你的精力应该几乎全花在前者**——读懂 03/04，把产品和提示词的判断做对，剩下的翻译工作大胆交给 Cursor。代码改坏了能重来，提示词和产品方向的判断才是这个岗位真正的护城河。
