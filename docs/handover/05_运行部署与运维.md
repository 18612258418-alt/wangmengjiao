## 一、本地运行（每次开发前）

⚠️ **不要用 `npm run dev` / `pnpm dev`**。因为本项目的 AI 能力依赖 `api/` 下的 Vercel Edge 函数，普通 dev server 不会启动它们，AI 功能会全部失效。

正确方式——用 Vercel CLI，它会同时启动前端 + 后端函数：

```bash
# 一次性：全局安装 Vercel CLI（如未装）
npm i -g vercel

# 一次性：安装项目依赖
pnpm install        # 或 npm install

# 每次开发：在项目根目录启动（会自动读取 .env.local）
vercel dev
```

启动后按提示在浏览器打开（通常 `http://localhost:3000`）。亲手批注一次、上传一张图、出一套题，确认 AI 都通。

> 如果 `vercel dev` 第一次要求登录/链接项目，按提示选已有的项目即可。

## 二、API Key 在哪（重要 · 涉及成本和安全）

### 本地：`.env.local`（项目根目录，已被 .gitignore 忽略，不会进 git）

里面有 4 个**服务端**变量（注意：**不带** `VITE_` 前缀，这样 Vite 不会把它们打包进前端 JS）：

| 变量名 | 含义 |
|--------|------|
| `DOUBAO_API_KEY` | 火山引擎 Ark（豆包）API Key |
| `DOUBAO_MODEL_ID` | 豆包视觉模型接入点 ID（形如 `ep-2026...`） |
| `DEEPSEEK_API_KEY` | DeepSeek API Key |
| `DEEPSEEK_MODEL_ID` | DeepSeek 模型 ID（如 `deepseek-v4-pro`） |

> 🔐 **安全红线**：这些是会产生真实费用的密钥。
> - 绝不要把 `.env.local` 提交到 git，绝不要贴到聊天/文档/截图里。
> - 交接时，密钥本身通过**安全渠道**（如密码管理器、私聊）单独给你，不写在任何文档里。
> - 怀疑泄露 → 立刻去对应平台控制台**重置 Key**，并更新本地 + Vercel 两处。

### 线上：Vercel 项目后台

线上不读 `.env.local`，读的是 Vercel 项目 Settings → Environment Variables 里配置的同名 4 个变量。**有两个 Vercel 项目，要分别配。**

## 三、两个环境（开发版 / 内测版）

这是为了"你天天改 main 时，不会把内测用户正在体验的版本搞乱"。

```
GitHub 仓库
├── main 分支             → Vercel 项目 A（开发版）   → 你自己用
└── release/demo-v1 分支   → Vercel 项目 B（内测版）   → 发给大学生
```

| 环境 | 分支 | 给谁 | 何时更新 |
|------|------|------|---------|
| 开发版 | `main` | 你自己验证 | 每次 push main 自动部署 |
| 内测版 | `release/demo-v1` | 内测大学生 | **只有**你主动合并到这个分支时才更新 |

> 关键配置：内测版 Vercel 项目的 **Production Branch 必须设成 `release/demo-v1`**（不是 main）。这一步如果设错，你 push main 内测地址就会跟着变。根目录 `docs/DEMO_DEPLOYMENT.md` 有创建项目、绑域名、DNS 的详细图文步骤（除环境变量名外仍可参考）。

## 四、日常发版流程

### 自己开发（不影响内测用户）
```bash
git checkout main
# ...改提示词 / 改代码...
git add .
git commit -m "feat: 描述这次改了什么"
git push origin main
# → 只有开发版自动部署，内测地址不变
```

### 给内测用户推新版本（每周五前）
```bash
git checkout release/demo-v1
git merge main              # 把本周开发成果合并过来
git push origin release/demo-v1
# → 只有内测版自动部署更新
git checkout main           # 切回 main 继续干活
```

> 合并前确保 main 在开发版上验过、是稳定的。发版后访问内测地址自测一遍再发给用户。

## 六、模型、超时与成本（运维要点）

### 超时是最常见的坑
Vercel 函数有网关时长限制（Edge 流式约 25 秒）。模型生成太久就会被切断 → 用户看到失败/占位内容。项目已做的应对：

| 措施 | 在哪 | 为什么 |
|------|------|--------|
| 关闭 DeepSeek 思考模式 | `api/deepseek.ts`、`api/import.ts`（`thinking: disabled`） | 思考模式又慢又容易超时，结构化任务不需要 |
| 出题用 `-flash` 不用 `-pro` | `GenerateDrawer.tsx` | flash 快，避免出题超时回退占位题 |
| 长内容走流式 | `streamText`/`callTextStreamed` | 首字节即下发，绕过网关整段超时 |
| 视觉/导入限 `max_tokens` + 压缩图片 | `api/doubao.ts`/`api/import.ts`/`api/...` | 控制单次生成时长压进 25s |
| 详情第2步 30s 兜底超时 | `App.tsx processImage()` | 超时就用第1步内容降级出卡，不让用户干等 |

> **如果内测反馈"经常生成失败/出占位内容"**：八成是超时。排查方向——是不是换了更慢的模型？是不是 max_tokens 调太大？是不是某步没走流式？

### 成本
- 两个模型都按调用量/token 计费。内测放量时建议在火山引擎、DeepSeek 控制台**看用量、设限额/告警**。
- 内测版和开发版**可以共用一套 Key**，但内测流量大时建议给内测版单独配 Key，便于单独监控和限流。
